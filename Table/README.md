## Table System

게임 데이터를 테이블 형태로 관리하기 위한 시스템입니다.  
데이터 정의, 로딩, 파싱, 직렬화, 쿼리까지 통합적으로 처리하도록 설계되었습니다.

---

### 개요

테이블 구조는 다음과 같이 구성됩니다.

- 테이블 클래스 ↔ 테이블 파일(XML)
- 내부 Record 클래스 ↔ 테이블의 한 행(Row)

즉, 데이터 구조를 코드로 직접 정의하고,  
테이블 데이터를 그대로 객체 구조로 매핑하도록 구현했습니다.

---

### 데이터 매핑 구조

- 테이블 클래스는 내부적으로 `Dictionary<Key, Record>` 형태로 데이터를 보유합니다.
- XML의 컬럼명과 Record 클래스의 필드명을 매칭하여 자동으로 값을 할당합니다.

---

### 특징

#### 1. 자동 매핑 시스템
- 클래스명 ↔ 파일명 매칭
- 필드명 ↔ 컬럼명 매칭
- 별도의 매핑 코드 없이 자동 데이터 바인딩

---

#### 2. 컬렉션 파싱 지원
XML에서 다음 타입을 직접 표현하고 파싱할 수 있도록 구현했습니다.

- Dictionary
- List
- Tuple

📌 방식
- 생성자 파라미터를 `,` 기준으로 파싱
- 자동으로 인스턴스 생성 후 Record에 할당

---

#### 3. 싱글톤 기반 관리
- 모든 테이블은 Singleton으로 관리
- 필요 시 자동 초기화

---

### 읽기 / 쓰기 시스템

#### ■ 읽기
- XML → 파싱 → 객체 생성

---

#### ■ 쓰기 (최적화 기능)

- 현재 메모리에 올라간 데이터를 **바이트 코드로 직렬화**
- 이후 로딩 시 XML 대신 바이트 파일을 직접 읽음

👉 효과
- XML 파싱 과정 제거
- 로딩 성능 향상

---

### 테이블 타입 분류

#### ■ System Table
- 시스템 부팅 이전에 로드되는 테이블
- 에셋 번들 외부에 존재하지 않는 데이터

📌 예시
- 리소스 로드 방식
- 시스템 설정값

---

#### ■ Game Table
- 일반 게임 데이터 테이블
- 에셋 번들 또는 빌드 포함 데이터

---

#### ■ Optional Table
- 자동 로딩되지 않는 테이블
- 필요 시 수동으로 로드

📌 예시
- 패치 리스트
- 특정 이벤트 데이터

---

#### ■ Editor Table
- Unity Editor 전용 테이블

---

### Multi Table 시스템

여러 테이블을 하나의 테이블처럼 사용할 수 있는 구조입니다.

---

#### 특징

- 동일한 Key 타입을 가지는 테이블들을 통합
- 하나의 인터페이스로 조회 가능

---

#### 제약 조건

- Key 타입 동일해야 함
- Key 중복 시 에러 발생

---

### Table MetaData

테이블 동작을 제어하는 메타데이터를 별도로 구성할 수 있습니다.

- 로딩 방식 제어
- 테이블 옵션 설정
- 커스터마이징 가능

---

### 설계 특징

#### 1. 데이터 중심 구조
- 코드 수정 없이 데이터 확장 가능

---

#### 2. 성능 최적화
- 바이트 직렬화 기반 로딩
- XML 파싱 제거

---

#### 3. 유연한 데이터 표현
- 컬렉션 타입 지원
- 복잡한 데이터 구조 표현 가능

---

#### 4. 계층적 테이블 구조
- System / Game / Optional / Editor 분리
- 로딩 타이밍 및 역할 분리

---

#### 5. 확장성
- 새로운 테이블 추가 시 구조 변경 최소화
- MultiTable로 데이터 통합 처리 가능

---
